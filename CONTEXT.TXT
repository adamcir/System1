Filesystem status:
- Shared `fs` module is active for all 3 outputs.
- Current root filesystem policy:
  - root `/` must mount from boot media
  - root is read-only
  - no normal RAM rootfs fallback
  - mount failure calls exactly `panic("Unable to mount FS")`
- VFS/block structure:
  - public shell API remains `fs_*`
  - `fs_core.c` dispatches through a mounted read-only driver
  - `block_core.*` provides the block-device boundary
  - `fat12_core.*` reads FAT12 root directories
  - `iso9660_core.*` reads ISO9660 root directories
- Boot media mapping:
  - `i386-floppy`: `scripts/mkimg-32.sh` copies `filesystem/fs32img` into `build/staging/rootfs-img32`, injects `KERNEL32.BIN`, `/boot/KERNEL.BIN`, `/boot/STAGE1.BIN`, and `/boot/STAGE2.BIN` into staging, then packs FAT12; stage2-loaded FAT12 data is exposed as the root block device
  - `iso-32`: `scripts/mkiso-i386.sh` copies `filesystem/fs32` into `build/staging/rootfs-iso32`, injects `/boot/kernel32.elf`, builds `rootfs.iso` from that staging tree, then adds `/boot/rootfs.iso` to the same staging tree; `grub-mkrescue` packs that staging tree directly, so GRUB reads `/boot/grub/grub.cfg` from the filesystem source
  - `iso-64`: `scripts/mkiso-x86_64.sh` copies `filesystem/fs64` into `build/staging/rootfs-iso64`, injects `/boot/kernel64.elf`, builds `rootfs.iso` from that staging tree, then adds `/boot/rootfs.iso` to the same staging tree; `grub-mkrescue` packs that staging tree directly, so GRUB reads `/boot/grub/grub.cfg` from the filesystem source
- Legacy RAM rootfs:
  - split into `ramfs_core.*`
  - retained as internal code, but no longer used as normal boot fallback
- Root image contents:
  - filesystem layout is edited in `filesystem/`, not generated from scratch by build scripts
  - build scripts must not write generated kernel files back into `filesystem/`; generated files belong only in `build/staging/*`
  - ISO filesystem sources own `/boot/grub/grub.cfg`; `boot/grub/*/grub.cfg` is no longer the ISO product source
  - baseline root folders follow Linux-style layout: `/bin`, `/boot`, `/dev`, `/etc`, `/home`, `/lib`, `/mnt`, `/opt`, `/proc`, `/root`, `/sbin`, `/sys`, `/tmp`, `/usr`, `/var`
- Path behavior:
  - absolute and relative paths supported
  - `.` and `..` supported
  - `mkdir` returns read-only filesystem
- Public API:
  - `fs_init(void)`
  - `fs_set_boot_context(uint32_t boot_magic, uint32_t boot_info_ptr)`
  - `fs_get_cwd_path(void)`
  - `fs_change_dir(const char* path)`
  - `fs_make_dir(const char* path)`
  - `fs_list_dir(const char* path, fs_dirent_t* entries, uint32_t cap, uint32_t* out_count)`

VGA status:
- Hardware cursor sync active (0x3D4/0x3D5)
- Hardware cursor visibility is explicitly enabled in `vga_core_init()` (scanline 14..15)
- Scroll-up on overflow enabled
- Color list exposed in `vga.h` via `#define VGA_COLOR_*` (0..15), with enum aliases.
- Compatibility alias for yellow is present (`VGA_COLOR_YELLOW`, `YELLOW`).
- Text editing API:
  - vga_text_begin(...)
  - vga_text_putc(...)
  - vga_text_backspace()
  - vga_text_left()
  - vga_text_right()
  - vga_text_delete()
  - vga_text_toggle_insert()
- Behavior:
  - right arrow bounded by text model
  - Enter in middle splits text to next line
  - Insert toggles insert/overwrite mode
  - `Home`/`End` APIs exist in VGA module, but are not bound in TTY key handling

Keyboard status:
- IRQ1 path active, polling fallback optional
- Default lock state after init:
  - `CapsLock=off`
  - `NumLock=off`
  - keyboard LEDs off (PS/2 LED sync)
- Supported editor control keys:
  - Left, Right
  - Insert, Delete
  - Home, End (currently not bound to TTY actions)
  - Tab
  - Caps Lock (letters-only, `Shift XOR CapsLock`)
  - Numpad numeric: `0-9`, `.`, `+`, `-`, `*`, `/`, Enter
  - Numpad navigation mode (with NumLock/Shift inversion):
    - Left, Right, Insert
  - Numpad `7` and `1` stay numeric (no Home/End mapping)
  - Ctrl+Alt+Del
- Key constants include:
  - KEY_LEFT/KEY_RIGHT
  - KEY_INSERT/KEY_DELETE
  - KEY_CTRL_ALT_DEL
- Numpad mode behavior:
  - number mode = `NumLock XOR Shift`
  - with `NumLock=off`, numpad numbers are blocked
  - numpad decimal (`.`) is always emitted as `.` (independent of `NumLock`)
  - real HW confirmation: numpad digits do not type when `NumLock` is off
